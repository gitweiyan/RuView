设计原则 & 参考来源
结合业界三大产品设计语言，提炼适用于本项目的设计规范
🏠
房间优先导航
参考：Vayyar Care + Apple Home
用户第一眼看到的是「哪个房间有人、状态如何」，而不是「某个人的指标」。这符合实际使用场景：家人查看的是「家里情况」，而不是盯着某个人的数字。
👤
匿名优先，可选识别
参考：Vayyar 隐私设计
WiFi CSI 默认检测「有几个人」及其体征，不强制绑定身份。家庭成员可选择性配对个人设备（手机/腕带），配对后才显示姓名，保护老人隐私。
🎯
告警精确到「房间+人」
参考：Vayyar 房间级告警
告警信息格式为「客厅 · 奶奶（人员A）· 血压偏高」，让家人第一时间知道在哪里、谁有问题，而不是泛泛的「有人异常」。
📊
「一览+下钻」信息层级
参考：医院病房管理系统
首页只显示房间状态卡（绿/黄/红），点击进入看详细体征。历史趋势先选房间再看数据。三层信息层级：房间总览 → 房间详情 → 个人指标历史。
👨‍👩‍👧
家庭共享，独立通知
参考：Care Daily Trusted Circle
多个家庭成员同时查看同一房间数据，各自独立配置自己的通知偏好（如：女儿全部告警，儿子只接紧急告警），通过「成员管理」统一控制。
🌙
深色主题 + 大字体
参考：RuView Foundation Book
沿用 RuView 的深色主题和专业 UI 语言，同时适配老人直接查看时的无障碍需求：关键数字不小于 16px，状态用颜色+图标双重编码。

🔔 告警推送策略
  • 紧急（跌倒/极低 SpO₂）：全员立即推送，振动 + 声音，15s 后自动呼叫第一联系人
  • 警告（血压/心率异常）：推送给配置了此类告警的成员，App 内高亮
  • 通知（服药/睡眠报告）：仅 App 内气泡，不推送打扰
  • 勿扰时间（22:00-7:00）：只有紧急告警突破勿扰
👤 多人识别规则
  • WiFi CSI 检测到 N 人 → 显示「人员 A/B/C…」匿名标识
  • 已配对设备（手机/腕带）在场 → 自动关联到姓名，显示头像
  • 未配对者保持匿名，体征数据仍然显示
  • 隐私模式：可在设置中关闭所有身份识别，一律匿名
📱 家庭成员权限等级
  • 「管理员」（通常是主要照料者）：全部数据 + 配置权限
  • 「查看者」：只读查看所有房间数据，可接收告警
  • 「仅紧急」：只接收紧急告警推送，无 App 查看权限
  • 邀请方式：分享链接或二维码，无需注册账号
🏠 房间配置流程
  • 首次设置：扫描 ESP32 → 命名房间 → 拖拽到户型图
  • 每个房间可配置：房间类型 / 常驻人员 / 特定告警阈值
  • 卫生间特殊配置：跌倒检测灵敏度 ↑，隐私级别 ↑
  • 一个房间支持多个 CSI 节点（覆盖盲区）
📊 趋势数据维度
  • 按「房间」筛选（而非按人，因为同一房间=同一 CSI 数据源）
  • 可切换：日视图 / 周视图 / 月视图
  • 指标：心率均值、睡眠时长、活动时间、告警次数、跌倒次数
  • 异常标记：自动在趋势图上标注告警发生时间点
🆘 紧急 SOS 流程
  • 跌倒自动触发 → App 全屏红色弹窗 + 计时 30s
  • 30s 内无人确认 → 自动拨打第一紧急联系人
  • 手动 SOS：首页长按任意房间卡 → 快速拨号
  • 呼叫记录保存在告警历史中
⏳ 等待你确认的问题
以下设计选项需要你的决策，再开始编码
Q1 · 平台选择
做成 手机 App（React Native），还是继续做 网页版（H5/PWA），还是两者都要？当前设计稿按手机 App 设计，但网页版也可以完全实现同样的设计。
Q2 · 多人体征显示方式
房间有 2+ 人时：方案A—每人一张卡片（如设计稿所示）；方案B—聚合显示均值（如「客厅平均心率 70 bpm」），可展开看各人；哪个更符合你的使用场景？
Q3 · 血压数据来源
WiFi CSI 目前能稳定检测心率、呼吸率、跌倒、睡眠、活动。血压需要可穿戴设备（腕带/血压计）手动同步。是否需要在 UI 中保留血压模块，并明确标注「需绑定设备」？
Q4 · 服务端/离线模式
数据是否需要云端备份（历史数据远程查看），还是纯本地 WiFi 内局域网访问即可（离开家就无法查看历史）？这影响系统架构设计。
Q5 · 设计语言偏好
当前设计延续 RuView 的深色科技感风格。是否需要调整为更温暖亲切的风格（适合老人和家庭感），例如浅色主题、更圆润的卡片、更大的图标？
